基于当前 Ripple 项目的架构说明

从一次请求讲清 Agent 系统如何运转

这份文档按模块拆解 Ripple 的 Agent 架构：HTTP 入口、Session 运行态、消息模型、主循环、 工具编排、定时任务、权限暂停、上下文压缩、Skill、沙箱、Fork 子代理以及前端事件协议。

Python 3.13+ FastAPI OpenAI-compatible API Tool Use Scheduled Jobs User Sandbox

系统总览

Ripple 的核心不是“调用一次模型”，而是一个可恢复、可暂停、可执行工具、可压缩上下文的 agentic loop。 系统边界可以拆成四条线：请求进入、模型推理、工具副作用、运行态持久化。

入口层

FastAPI 接收 OpenAI 兼容的 chat completions 请求，解析 user input、system prompt、model、 stream 模式和 session id。

运行态

Session 持有消息历史、模型历史、工具上下文、权限状态、当前任务、token 统计和 compactor 状态。

主循环

Agent Loop 负责调用模型、识别 tool use、执行工具、回灌 tool result，并决定继续下一轮还是终止。

工具层

工具统一继承 Tool，由 orchestration 层处理去重、权限、串并行、结果截断和上下文修改。

能力层

Skill 把领域工作流包装成模型可调用的工具；AgentTool 把复杂子任务 fork 到后台 agent 执行。

调度层

Scheduler 以 user sandbox 为边界保存定时任务，支持延迟提醒、周期任务、沙箱命令和后台 agent prompt。

隔离层

Sandbox 以 user_id 为隔离单位，workspace 长期存在，session 只是该 user 下的运行态。

核心判断 Ripple 的架构重点是“模型可以提出动作，但动作必须经过系统执行边界”。因此 tool use 不是直接副作用， 而是先进入工具编排层，再经过权限、沙箱、锁、结果标准化，最后才以 tool result 的形式回到模型上下文。

一次请求路径

从用户发送一条消息到浏览器收到流式输出，中间会经过 Server、Session、SSE 适配层、Agent Loop、 LLM Client 和 Tool Orchestration。下面是当前系统中的主要调用路径。

1
POST /v1/chat/completions routes.py 解析请求，提取最后一条 user message、调用方 system prompt、 model、max_turns、stream 参数，并通过 X-Ripple-User-Id 取得 user_id。
2
SessionManager.get_or_create_session() 如果请求带 session_id，优先从内存取；取不到则从磁盘恢复；仍不存在时创建新 session， 并初始化 sandbox workspace、runtime dir、ToolUseContext 和 LLM client。
3
stream_query_as_sse() 或 collect_query_response() SSE 层把 agent loop 的内部事件转换成 OpenAI 兼容 chunk，同时附加 Ripple 扩展事件： tool_call、tool_result、task_progress、agent_stop、heartbeat。
4
query() 构造消息上下文 合并 system prompt、历史 model messages 和本轮 user input，然后调用 query_loop()。
5
query_loop() 多轮执行 每轮调用模型流，收集 assistant message，检测 tool use，执行工具并把 tool result 写回消息历史。 如果还有 tool result 需要模型理解，则进入下一轮。
理解这条路径的关键 Server 只负责接入请求和维护 session；SSE 层负责把内部事件翻译给前端； Agent Loop 负责持续推进任务；工具层负责执行真实副作用。每一层只处理自己的边界问题， 因此系统可以在等待用户确认、上下文过长、工具失败或请求取消时保持可恢复。

Server 与 Session

Server 层负责把 HTTP 请求变成一个可恢复的 Agent 运行态。这里最关键的不是路由本身， 而是 session 如何绑定 user sandbox、工具上下文、权限状态和持久化消息。

  • src/interfaces/server/routes.py：HTTP 端点、流式/非流式聊天、权限恢复。
  • src/interfaces/server/sessions.py：Session 数据结构、创建、恢复、挂起、TTL 清理。
  • src/interfaces/server/sse.py：Agent 内部事件到 SSE / OpenAI chunk 的转换。
  • src/interfaces/server/deps.py：从 header 解析 X-Ripple-User-Id。

Session 保存什么

字段 作用
messages 完整展示历史，面向前端和持久化。这里尽量保留用户可见信息。
model_messages 模型上下文历史，可能经过 compact 或清理，用于下一轮模型输入。
context ToolUseContext，工具执行需要的 cwd、workspace、权限、sandbox、当前消息等。
status idle、running、awaiting_user_input、awaiting_permission。
pending_permission_request 危险工具被拦截时保存待确认请求，前端确认后重放原工具调用。
context_manager 封装 compactor 状态，session 挂起后可恢复压缩参数和估算状态。

创建 session 时发生了什么

新 session 会先解析模型别名和最大轮数，然后建立一个外部可见的 session id，以及一个内部工具上下文使用的 session id。如果启用了 sandbox，系统会确保当前 user 的 workspace 存在，并为本次 session 创建独立的运行时目录。 随后，SessionManager 会创建 ToolUseContext 和 LLM client，并把它们绑定到 session。

这样设计的原因是：会话、模型、工具、权限和沙箱并不是彼此独立的全局状态，而是一组必须共同恢复的运行态。 当 session 从磁盘恢复时，只要这些字段能够重新构造，Agent 就可以继续沿用同一条对话历史运行。

设计要点 Session id 是一次对话的运行态标识；user id 才是沙箱和 workspace 的隔离维度。 同一个 user 的多个 session 可以共享 workspace，但工具执行需要通过 user 级锁避免互相踩写。

Message 模型

Ripple 内部消息模型的目标是统一 provider 差异。模型 API 可以是 OpenAI 兼容格式， 前端可以消费 SSE，但 Agent Loop 内部只处理一组稳定的 Pydantic message 类型。

  • src/ripple/messages/types.py：核心消息类型定义。
  • src/ripple/messages/utils.py：创建、序列化、反序列化、提取 tool use。
  • src/ripple/messages/cleanup.py：模型上下文清理。
类型 用途 典型内容
SystemMessage 系统提示或系统通知。 { type, content, level }
UserMessage 用户输入、tool result、meta 提示。 message.content 可以包含 text 或 tool_result block。
AssistantMessage 模型输出。 message.content 可以包含 text 和 tool_use block。
StreamEvent 实时文本事件，不一定进入最终消息历史。 stream_start、stream_chunk、stream_end、heartbeat。
AgentStopEvent Agent loop 主动暂停。 ask_user、permission_request、max_turns。

Tool use 与 tool result 的配对

模型发出的工具调用是 assistant message 里的 tool_use block；工具执行结果会被包装成 user message 里的 tool_result block，并通过 tool_use_id 关联回原调用。 下一轮模型看到的是“我刚才调用了工具，并收到了这个结果”。

为什么要配对 工具调用和工具结果必须稳定配对，否则模型无法判断某个结果对应哪一次动作。 tool_use_id 就是这条链路的关联键。它让系统可以支持多个工具调用、并发执行、 权限等待后重放，以及前端展示工具调用状态。
两套历史 session.messages 偏向产品展示和完整追溯； session.model_messages 偏向模型可承受的上下文。压缩和清理通常影响后者， 但不应该破坏前端展示历史。

Agent Loop

query_loop() 是 Ripple 的核心。它不是一个简单的 while 调模型，而是一个带状态转移、 错误恢复、上下文压缩、工具抢跑和停止事件的执行引擎。

  • src/ripple/core/agent_loop.py：主循环。
  • src/ripple/core/state.py：QueryState。
  • src/ripple/core/query_params.py：一次 query 的入参。
  • src/ripple/core/transitions.py：继续和终止状态。
  • src/ripple/core/errors.py：错误分类和 token gap 解析。

状态对象很薄，但边界清楚

QueryState 不承担业务逻辑，只保存循环推进所需的最小状态：当前模型上下文、 工具上下文、轮数和上一轮状态转移原因。它的价值在于让每一次循环都能明确回答三个问题： 模型现在能看到什么、工具现在能访问什么、循环下一步应该继续还是终止。

主循环阶段

0
中止检查 如果 AbortSignal 已经被设置，当前 query 直接返回，不继续调用模型或工具。
1
轻量清理和主动压缩 先执行 lightweight_cleanup() 清理旧 tool_result / tool_input，再用 should_compact() 判断是否需要 LLM summary 或截断。
2
任务提醒和 current messages 同步 如存在 Task reminder，则追加 meta message；随后把当前 messages 同步到 ToolUseContext.current_messages，供 AgentTool fork 使用。
3
调用模型流 client.stream() 产出 StreamEvent 和 AssistantMessage。 文本 chunk 会直接传给上层，完整 assistant message 用于收集 tool use。
4
检测工具调用 extract_tool_use_blocks() 从 assistant message 提取 tool_use。 并发安全工具会交给 StreamingToolExecutor 尝试抢跑。
5
执行剩余工具 流式抢跑已接管的 tool id 会跳过；剩余工具进入 run_tools()， 根据并发安全性分批执行。
6
决定继续或停止 没有 tool use 时执行 stop hooks；有 tool result 时进入下一轮。若工具请求暂停、 达到 max_turns 或遇到不可恢复错误，则产生 AgentStopEvent 或终止。

循环为什么要分阶段

Agent Loop 的核心难点不是调用模型，而是把一次模型输出放进可控的生命周期里。 调模型之前要先确认上下文仍然可用，避免过长历史导致请求失败；调模型期间要持续把文本流推给上层， 同时识别工具意图；模型结束后要把工具结果作为新的事实写回上下文；如果工具产生了暂停请求， 循环必须停在可恢复的位置。

这种分阶段结构让系统可以处理真实工程中的边界情况：用户中止请求、模型输出过长、上下文超限、 工具需要确认、工具结果很大、工具可以并发执行、以及最大轮数保护。每个边界都有明确的位置处理， 不会混在一段不可恢复的调用链里。

为什么 tool result 需要下一轮 工具执行结果不是直接拼进最终回答。模型必须在下一轮看到 tool result，然后决定继续调用工具还是输出最终答案。 这就是 agent loop 与简化 function call 示例的核心差异。

流式 API 适配

API 层把 provider 的流式响应变成 Ripple 的内部事件。它需要同时处理文本流、reasoning、 tool call 参数增量、usage token，以及 finish_reason 缺失的 fallback。

  • src/ripple/api/client.py：统一 LLMClient 入口和 create_client。
  • src/ripple/api/streaming.py：把 OpenAI SDK stream 转成 StreamEvent / AssistantMessage。
  • src/ripple/api/openrouter.py：OpenRouter/OpenAI 兼容实现。
  • src/ripple/api/anthropic.py：Anthropic 适配。

流式处理的三个缓冲区

缓冲区 作用
current_text 累计文本 delta，最终构造成 assistant text block；每个 delta 同时以 stream_chunk 推给前端。
tool_calls_map 按 tool call index 累计 id、name 和 arguments buffer，finish 时再 JSON parse。
last_usage 保存 usage token，供 compactor 校准和 SSE usage 汇总。

Reasoning 的处理

有些 provider 把思考内容放在 delta.reasoning 或 delta.reasoning_content。 当前实现会把它包装成 <think>...</think> 文本块，复用前端 MarkdownRenderer 的折叠渲染能力， 不额外增加协议类型。

工具参数不是一次性到达

OpenAI 兼容流式 tool call 的 function.arguments 会分多段到达，所以 process_stream_response() 先累积 args_buffer，直到 finish_reason 才执行 json.loads()。如果解析失败，tool_use block 会带上 _args_parse_error， 后续工具层会返回结构化错误，而不是直接崩掉主循环。

工具系统

工具系统的重点是可控执行。模型只产生 tool_use 意图；真正的副作用由工具编排层决定是否执行、 何时执行、串行还是并行、是否需要用户确认，以及如何把结果回灌给模型。

  • src/ripple/tools/base.py：Tool、ToolResult、风险等级和 schema 转换。
  • src/ripple/tools/orchestration.py：查找工具、去重、分批、执行、权限、结果消息。
  • src/ripple/tools/streaming_executor.py：流式期间抢跑并发安全工具。
  • src/ripple/tools/builtin/：Bash、Read、Write、Skill、Agent、AskUser、Task、Schedule 等内置工具。

Tool 抽象解决什么问题

工具抽象把“模型可以调用什么”和“系统如何安全执行”放在同一个协议里。 每个工具都需要声明名称、描述、输入结构、风险等级、最大结果大小，以及是否可以并发执行。 这些信息同时服务于三件事：生成模型可理解的 tool schema、执行前做权限判断、执行后把结果标准化为 tool result。

这样做的原因是，工具不是普通函数。它们可能读写文件、启动命令、访问外部服务或改变 session 状态。 如果没有统一抽象，Agent Loop 就无法在不同工具之间稳定处理权限、日志、并发和错误。

工具执行细节

1
去重 _dedup_tool_calls() 用 name + input 生成稳定 key， 相同调用只执行第一个，其余返回“重复调用已跳过”的结果。
2
分批 _partition_tool_calls() 调用每个工具的 is_concurrency_safe()。 连续安全工具合并成并发批次，不安全工具单独串行执行。
3
权限检查 PermissionManager.check_permission() 在真正执行前运行。 如果需要确认，会返回 permission_request 并暂停 agent loop。
4
调用工具 tool.call() 返回 ToolResult。 如果 data 是 Pydantic model，则用 model_dump_json() 序列化。
5
结果标准化 结果超过 max_result_size_chars 会截断；随后创建 tool_result message，附带 tool_use_id、is_error 和 source assistant uuid。

StreamingToolExecutor 的分工

普通 run_tools() 要等模型完整响应结束后才能执行工具。 StreamingToolExecutor 则在流式过程中一旦看到并发安全 tool_use，就立即 asyncio.create_task() 执行。这样搜索、读文件、AgentTool 这类安全工具可以与模型剩余输出重叠， 减少端到端等待时间。

它不会接管所有工具，只接管明确声明为并发安全的工具。模型流结束后，主循环会跳过已经被抢跑的工具， 只把剩余工具交给普通编排流程。这样既能利用并行能力，又不会让写文件、改凭证、等待用户确认等操作提前发生。

工具不是信任边界内的自由执行 任何可能修改文件、执行命令、发外部请求或启动子代理的能力，都必须显式声明风险等级，并走权限、 沙箱和日志摘要。模型输出的参数只能被当作待验证输入。

定时任务

定时任务把“稍后执行”和“周期执行”纳入同一套 user sandbox 边界。它不是让模型自己 sleep 或轮询， 而是由 Server 进程内的 SchedulerManager 负责扫描、持久化、执行和记录 run history。

  • src/ripple/scheduler/models.py：ScheduledJob、ScheduledJobState、ScheduledRun 数据模型。
  • src/ripple/scheduler/store.py：按 user sandbox 保存 jobs、jobs-state 和 runs。
  • src/ripple/scheduler/manager.py：后台 tick、并发保护、运行状态、重启恢复。
  • src/ripple/tools/builtin/schedule.py：模型可调用的 Schedule 工具。
  • src/interfaces/server/scheduler_agent.py：定时 agent job 的隔离运行入口。
  • src/interfaces/web/src/components/ScheduledTasksPanel.tsx：前端 Scheduled Tasks 面板。

任务类型

维度 当前取值 行为
execution_type command 通过 run_sandbox_command() 在当前 user 的 nsjail sandbox 中执行 shell 命令。
execution_type agent 为本次 run 创建独立 runtime，运行一次 agent loop，并把最终摘要写入 run 记录。
schedule_type once 按 run_at 或 delay_seconds 执行一次，完成后不再计算下一次。
schedule_type interval 按 interval_seconds 计算下一次运行时间，可用 max_runs 限制次数。

调度路径

1
创建或更新 job REST API 或 Schedule 工具构造 ScheduledJob。 Server 不接受 body 里的 user_id，而是使用请求 header 解析出的当前 user。
2
持久化定义和状态 job definition 写入 scheduled-tasks/jobs.json； runtime state 写入 scheduled-tasks/jobs-state.json， 避免频繁变化的运行状态污染用户可编辑定义。
3
后台 tick 扫描 SchedulerManager 默认每 5 秒扫描所有 user sandbox， 只启动 enabled、到期且没有 running marker 的任务。
4
写入 running marker 在 _state_lock 内创建 ScheduledRun， 持久化 running_at/current_run_id，再释放锁执行真实任务。
5
执行与收口 command job 使用 user 级 sandbox lock；agent job 使用独立 sched-* runtime。 完成后写 run 终态、duration、stdout/stderr tail 或 agent summary，并计算下一次运行。

API 与工具入口

入口 用途
POST /v1/sandbox/schedules 创建定时任务，支持 once / interval、command / agent。
GET /v1/sandbox/schedules 列出当前 user 的任务摘要。
PATCH /v1/sandbox/schedules/{job_id} 更新任务定义和 enabled 状态，并重新计算 next_run_at。
POST /v1/sandbox/schedules/{job_id}/run 立即触发一次任务；如果任务正在运行，会按 running marker 跳过。
GET /v1/sandbox/schedules/{job_id}/runs 读取 run history，前端面板用它展示 running、success、failed、timeout 等状态。
可靠性边界 Scheduler 启动时会恢复被 server 重启打断的 running run，把它们标记为 failed，并清空 running marker。 这让定时任务不会因为进程退出永久卡在运行中。
当前范围 当前实现覆盖 once / interval。cron、timezone、本地 wall-clock 语义、失败退避和主动投递到原 chat 仍属于后续演进项；这些方向已在 docs/SCHEDULER_DESIGN.md 中约束。

权限与暂停

Ripple 把“等待用户确认”建模成 agent loop 的正常暂停状态，而不是异常。 这样前端可以展示确认 UI，用户确认后 server 能重放原工具调用并更新同一条 session 历史。

  • src/ripple/permissions/manager.py：权限 key、一次性授权、session 授权、permission request。
  • src/ripple/permissions/levels.py：权限模式和工具风险等级。
  • src/interfaces/server/routes.py：/permissions/resolve 端点重放工具。
  • src/ripple/tools/builtin/ask_user.py：模型主动向用户提问的暂停工具。

权限检查如何影响循环

权限检查发生在工具真正执行之前。如果操作已被允许，工具继续执行；如果操作需要确认， 工具层会先生成一条“等待用户授权”的 tool result，再向 Agent Loop 返回暂停信号。 这条暂停信号会被转换成 AgentStopEvent，前端据此展示确认界面。

这种设计避免了两类问题：一是危险操作在用户确认前已经发生；二是系统把等待确认当成错误处理。 在 Ripple 里，等待确认是可恢复状态，仍然属于正常执行路径。

确认后的恢复

前端调用 POST /v1/sessions/{session_id}/permissions/resolve。如果用户选择 allow 或 always， server 会给 PermissionManager 授权，再用保存的 tool_use_id、tool name 和 params 重新调用 execute_tool()。随后 _replace_tool_result() 会把之前的等待占位结果替换成真实结果。

状态 触发来源 前端应该做什么
awaiting_permission 危险工具需要确认。 展示 allow / always / deny，并调用 resolve 端点。
awaiting_user_input AskUser 工具或类似用户输入需求。 展示问题和选项，等待用户下一条消息。
idle 完成、取消、确认处理完毕。 允许继续发起下一轮请求。

上下文压缩

长任务里消息会不断增长。Ripple 的压缩不是单一策略，而是分层降级： 先做无损或低损清理，再尝试 LLM 摘要，最后在紧急情况下定向裁剪或硬截断。

  • src/ripple/compact/auto_compact.py：压缩协调器和触发判定。
  • src/ripple/compact/cleanup.py：轻量清理旧 tool_result / tool input。
  • src/ripple/compact/summary.py：LLM 摘要压缩。
  • src/ripple/compact/truncate.py：定向裁剪和硬截断。
  • src/ripple/compact/context_manager.py：server session 侧的模型上下文准备与 compactor 状态。

触发与校准

AutoCompactor.should_compact() 使用增量 token 估算，避免每轮都重算全部历史。 估算阈值默认是模型上下文窗口的 75%。模型返回真实 input_tokens 后， calibrate_with_api_tokens() 会扣掉 system prompt 和 tool definitions 的结构性 overhead， 用真实 token 反向校准估算缓存。

三层策略

策略 何时触发 特点
lightweight_cleanup 每轮开始，如果消息数增加。 不调用 LLM，清理旧工具大输出和旧工具输入，保留最近若干轮。
compact 估算 token 超过阈值。 优先 LLM summary；连续失败达到断路器阈值后直接 hard truncate。
reactive_compact API 明确返回 context too long。 如果能解析 token gap，先 targeted trim；否则减少保留轮数，进行更激进压缩。

Reactive compact 的目的不是追求最优摘要，而是在 API 已经拒绝请求时尽快回到可调用状态。 如果错误中能解析出超出的 token 数，系统会优先裁掉足够多的内容；如果无法精确判断， 就降低保留轮数并使用更激进的压缩策略。它是失败恢复机制，不是常规上下文管理的首选路径。

产品体验与模型上下文分离 压缩服务于模型调用，不应等同于删除用户历史。server 会从 context.current_messages 回写 model_messages， 让下一轮模型使用压缩后的上下文，同时保留 messages 给前端展示。

Skill 系统

Skill 是把领域工作流包装成模型可调用能力的层。它不是 Python 插件，而是带 YAML frontmatter 的 Markdown 指令， 由 SkillTool 暴露给模型。

  • src/ripple/skills/types.py：Skill 数据结构、参数替换、allowed tools 判断。
  • src/ripple/skills/loader.py：shared / workspace 两级加载与缓存。
  • src/ripple/skills/skill_tool.py：模型调用的 Skill 工具。
  • src/ripple/skills/executor.py：inline 和 fork 两种执行方式。

加载层级

load_shared_skills() 从配置 skills.shared_dirs 读取共享技能； load_workspace_skills(workspace_root) 会先加载 shared，再加载 workspace/skills/，后者覆盖前者。缓存使用目录 mtime，避免每次工具调用都扫描磁盘。

Skill 文件入口

loader 只加载两类 Markdown：文件名是 SKILL.md，或 frontmatter 中包含 name / description。普通 README 或 references 文档不会误加载成 skill。

一个 Skill 通常包含名称、描述、可用工具、执行上下文和正文说明。正文并不是给用户看的说明书， 而是给模型看的操作规范：什么时候使用这个能力、按什么顺序调用工具、哪些动作必须确认、 失败时如何降级、最后应该如何汇报结果。

Inline 与 Fork

模式 实现 适合场景
inline execute_inline_skill() 把 skill 内容作为 tool result 返回给当前模型。 短流程、规则说明、领域命令手册。
fork execute_forked_skill() 通过 AgentTool 启动后台子 agent。 复杂、多步骤、可异步执行的领域任务。
Skill 的边界 Skill 只增加模型的操作知识和可用工具范围，不绕过权限系统。即便 skill 允许某个工具， 危险操作仍然要经过工具自身风险等级和 PermissionManager。

沙箱与持久化

Ripple 的沙箱以 user 为单位，而不是 session 为单位。这样用户工作区可以跨多轮、多 session 长期存在， 同时 session runtime 只保存对话状态、任务状态和后台输出。

  • src/ripple/sandbox/manager.py：user sandbox 生命周期、session 持久化、摘要。
  • src/ripple/sandbox/config.py：路径约定和环境状态检测。
  • src/ripple/sandbox/workspace.py：workspace 创建、销毁、大小统计。
  • src/ripple/sandbox/storage.py：messages.jsonl、meta.json、挂起状态读写。
  • src/ripple/sandbox/executor.py：nsjail 执行相关能力。

目录职责

运行时目录分成共享缓存、user sandbox 和 session runtime 三类。共享缓存用于复用包管理器缓存； user sandbox 保存长期 workspace、nsjail 配置、用户级凭证和 scheduled-tasks/； session runtime 保存某一次会话的消息、任务列表和后台任务输出。这个拆分让 workspace 可以长期存在， 同时允许 session 被挂起、恢复或清理。 

.ripple/sandboxes/<user_id>/
  workspace/
  credentials/
  scheduled-tasks/
    jobs.json
    jobs-state.json
    runs/<job_id>/<run_id>.json
  sessions/<session_id>/

user lock

SandboxManager.user_lock(user_id) 返回 user 级 asyncio.Lock。 所有会修改同一 user workspace 的工具调用，都应该在这个锁下运行。原因是多个 session 可以共享同一个 workspace， 如果 Bash、Write、CLI 登录等副作用并发执行，容易造成文件竞争或凭证状态不一致。

挂起与恢复

Session TTL 到期时，SessionManager._cleanup_expired() 会取消运行中的任务， 然后调用 _suspend_to_disk() 把 session 状态写入磁盘。恢复时 resume_session() 会读取状态、重建 ToolUseContext 和 LLM client，并恢复 compactor 状态。

default user 的特殊规则 SandboxManager.teardown_sandbox() 默认禁止销毁 default user sandbox。 这是管理端点层面的保护，避免误删默认工作区。

Subagent 与 Fork

Ripple 的 AgentTool 不是“再开一段聊天”，而是 fork 一个继承父上下文的后台 agent。 它可以把复杂子任务放到后台运行，并把输出写入 session runtime 的 task-outputs。

  • src/ripple/tools/builtin/agent_tool.py：AgentTool 输入、输出、fork 启动。
  • src/ripple/core/fork.py：fork 消息构造、防递归、cache 友好占位符。
  • src/ripple/core/background.py：后台任务注册和输出收集。
  • src/ripple/tasks/manager.py：TaskCreate/TaskUpdate 的 todo 管理，不要与后台任务混淆。

Fork 消息如何构造

build_forked_messages() 会保留父 assistant message 的完整 tool_use blocks， 然后构造一个 user message：先为这些 tool_use 填入统一占位 tool_result，再追加子任务指令。 这样多个 fork 子 agent 的请求前缀尽可能字节级一致，有利于 prompt cache。

Fork 子 agent 继承父 agent 的当前上下文，再追加一段专门的子任务指令。为了让多个子任务共享更稳定的 prompt 前缀，系统会给父 assistant message 中的工具调用补上统一占位结果，然后把真正不同的子任务指令放在最后。 这能减少上下文差异，也降低子 agent 误解父流程状态的概率。

防递归

is_in_fork_child() 会扫描历史消息中是否存在 <fork-boilerplate>。 如果子 agent 再次调用 AgentTool，工具会直接返回错误，要求它在当前进程内直接执行。

ToolUseContext 的继承

子 agent 会继承父上下文里的工具集合、cwd、workspace_root、sandbox_session_id、session_runtime_dir、 user_id、sandbox_manager、permission_manager 和 thinking 等字段。但它会生成自己的 session_id，并重置 read_file_state。

Hooks 与 Recovery

当模型没有继续 tool_use 时，主循环并不会马上结束。它还会执行 stop hooks，允许系统对“模型认为完成” 这个状态做最后验证。同时，API 错误会进入明确的恢复分支。

  • src/ripple/core/stop_hooks.py：模型停止后的 hook 处理。
  • src/ripple/core/hooks.py：hook 基础逻辑。
  • src/ripple/core/recovery.py：max_output_tokens 两阶段恢复。
  • src/ripple/core/errors.py：错误分类、连接错误重试、PTL token gap 解析。

Stop hooks 的位置

Stop hooks 只在模型没有继续请求工具时运行。也就是说，模型认为自己已经可以结束， 系统才进行最后检查。如果 hook 发现阻塞问题，可以把问题写回消息历史并让模型继续一轮； 如果 hook 明确阻止继续，循环就以对应终止状态结束。

这个位置很重要：hook 不应该打断正常工具执行，也不应该替代权限系统。它更像是完成前的系统校验， 用来补充模型自身无法可靠判断的规则。

max_output_tokens 恢复

ErrorRecovery 对 max output 错误采用两阶段策略：第一次先把 max_tokens 升级到默认值的 1.5 倍，不注入额外消息；如果仍失败，再注入 recovery user message， 要求模型直接续写并拆小剩余工作。恢复次数有上限，避免无限循环。

连接错误重试

连接类错误只有在本轮尚未产出任何 AssistantMessage 时才会重试。 如果已经产生 assistant message，再重试可能导致重复消息污染历史，所以会走终止错误路径。

前端事件协议

前端不只消费文本流，还要理解 Agent 的运行状态：工具调用、工具结果、任务进度、权限暂停、 用户输入暂停、定时任务状态和 heartbeat。SSE 层负责把内部事件转换为前端可以直接消费的事件。

  • src/interfaces/server/sse.py：SSE 事件生成。
  • src/interfaces/web/：Next.js 前端消费事件并渲染状态。
  • src/interfaces/server/schemas.py：请求、响应和 session 结构。
事件 来源 前端用途
OpenAI chunk StreamEvent.stream_chunk 实时渲染 assistant 文本。
tool_call AssistantMessage.tool_use 展示模型准备调用哪个工具和参数。
tool_result UserMessage.tool_result 展示工具完成状态，内容只截取前 500 字符。
task_created / task_updated Task 工具结果 维护 todo 进度条和任务列表。
Scheduled Tasks 轮询 /v1/sandbox/schedules 与 /runs 展示定时任务定义、下一次运行时间和最近 run 状态。
agent_stop AgentStopEvent 切换到等待用户输入、等待权限或 max_turns 确认状态。
heartbeat SSE wrapper 超时心跳 保持长连接活跃，避免前端误判请求结束。
SSE 层的额外职责 流结束后，SSE 层会把本轮 new messages 追加到 session.messages， 并从 context.current_messages 回写 session.model_messages。 因此它既是协议适配层，也是本轮 query 的持久化收口点。

扩展指南

如果要在当前系统里扩展能力，优先判断它是“新工具”还是“新 Skill”。工具适合新增可执行副作用； Skill 适合把已有工具编排成领域流程。

新增一个工具

1
创建工具类 在 src/ripple/tools/builtin/ 下实现 Tool[Input, Output]。
2
定义输入模型和 schema 用 Pydantic 模型描述参数；必要时重写 _get_parameters_schema()。
3
声明风险等级 修改文件、执行命令、发消息、删除资源等操作不要标为 safe。
4
实现并发安全判断 只读、无共享状态的工具可以返回 true；会写 workspace 或改凭证的工具应返回 false。
5
注册到 server 工具列表 把实例加入 sessions.py::_get_server_tools()。
6
补测试和日志摘要 覆盖成功、权限拒绝、异常、长输出截断；必要时重写 log_input_summary()。

新增一个 Skill

如果能力不需要新增底层副作用，只是希望模型按某个领域流程使用现有工具，则创建 SKILL.md 更合适。放在 shared skills 目录可全局可见；放在 workspace/skills/ 只影响当前 user workspace。

选择工具还是 Skill 如果需要系统新增一种实际能力，例如调用新的外部 API、读写一种新资源或执行一种新副作用，应新增工具。 如果只是把已有工具组织成稳定流程，例如某类文档处理、某个 CLI 的使用规范或某个业务步骤，应新增 Skill。

本文档对应当前仓库模块结构生成；如果后续调整了 agent loop、工具协议或 sandbox 生命周期， 应同步更新对应章节，而不是只改概念图。
